# AI 辅助多方向简历生成系统

## 项目概览

**项目类型：** 个人技术门户扩展项目  
**项目角色：** 方案设计与原型开发  
**技术栈：** Markdown、YAML Front Matter、Node.js、JSON、Hexo、AI Prompt  
**项目定位：** 将项目经历内容转化为可复用的多方向简历材料

本项目围绕个人技术门户中的项目经历和简历模块展开，目标是把已经沉淀的项目 Markdown 内容进一步结构化，并探索通过 AI 或本地规则生成不同方向的在线简历页面。

## 项目背景

个人项目内容通常会重复出现在多个地方：博客文章、项目经历页、在线简历、求职材料和课程验收说明。如果每个地方都手动维护，后续很容易出现内容不一致。

因此，本项目尝试把项目经历作为统一数据源，通过结构化字段标注项目方向、技术栈、权重和角色，再根据不同展示目标生成简历内容。

## 核心目标

- 将项目页从普通 Markdown 扩展为可被程序识别的内容数据源。
- 使用 front matter 标注项目是否进入简历、适合方向、技术栈和排序权重。
- 设计“项目内容 -> 结构化 JSON -> 多方向简历页面”的生成流程。
- 在 AI 不可用时保留本地规则降级，保证站点构建稳定。
- 为 Web、C++ / 图形学、通用软件开发等方向维护不同简历版本。

## 内容结构设计

项目页 front matter 中加入与简历生成相关的字段：

```yaml
resume: true
directions:
  - web
  - software-engineering
tech_stack:
  - Hexo
  - GitHub Actions
  - Markdown
role: 独立开发
weight: 90
```

正文仍然保持适合阅读的项目详情结构：

```text
项目概览
项目背景
核心功能
主要工作
技术实现
问题与解决方案
项目亮点
项目成果
个人收获
```

这样既不牺牲页面可读性，也为自动化提取提供了稳定输入。

## 生成流程

设计中的生成流程如下：

```text
读取 source/projects/**/index.md
-> 解析 front matter
-> 判断 resume 字段
-> 提取标题、角色、技术栈和正文摘要
-> 按 directions 分组
-> 按 weight 排序
-> 生成多方向简历数据
-> 写入 resume 页面或中间 JSON
```

AI 可以参与“正文摘要”和“简历表述优化”环节。本地规则负责兜底，保证没有 API Key 时仍然能得到可用结果。

## AI 参与方式

AI 不直接凭空生成经历，而是基于已有项目页做提炼：

- 将长篇项目复盘压缩为 2 到 4 条简历要点。
- 根据目标方向选择更相关的技术关键词。
- 将叙事型文字转化为更适合简历的成果表达。
- 检查是否存在过度夸张或事实不明确的描述。

这种方式比让 AI 从零写简历更可靠，因为输入材料来自真实项目。

## 本地降级策略

为了避免构建依赖外部 API，项目设计了本地降级策略：

- 如果存在 API Key，则调用 AI 生成更自然的摘要。
- 如果 API 调用失败，则读取项目概览和核心功能生成基础要点。
- 如果某些字段缺失，则使用默认值并在构建日志中提示。
- 简历生成失败不应阻塞 Hexo 站点的基础构建。

这个策略的核心是稳定性优先。AI 能增强内容质量，但不能成为站点可用性的单点依赖。

## 项目亮点

- 将项目经历、博客复盘和简历展示统一纳入同一套内容源。
- 使用 front matter 为 Markdown 内容补充结构化语义。
- 设计了可降级的 AI 辅助生成流程，兼顾自动化和稳定性。
- 支持按方向生成不同简历版本，减少重复维护成本。
- 保留人工复核入口，避免 AI 生成内容脱离真实项目。

## 项目成果

- 明确了项目页作为简历数据源的字段规范。
- 在项目经历页面中加入 `resume`、`directions`、`tech_stack`、`weight` 等字段。
- 整理出多方向简历页面结构，为后续自动生成提供承载位置。
- 形成了 AI 简历生成流程设计文章：[AI 辅助简历生成流程设计：从项目复盘到方向化简历](/2026/06/01/AI-辅助简历生成流程设计/)。

## 个人收获

这个项目让我更清楚地理解了内容工程和自动化之间的关系。简历不是孤立文档，而是可以从项目材料中提炼出来的展示结果。

相比直接维护多份简历，把项目经历结构化之后再生成不同方向版本，更适合长期维护，也更符合个人技术门户的定位。
