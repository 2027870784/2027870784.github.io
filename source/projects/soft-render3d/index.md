# 从零构建 C++ 3D 软光栅渲染器

## 项目概览

**项目类型：** 计算机图形学个人项目  
**项目角色：** 独立开发  
**技术栈：** C++、CMake、MVP 变换、重心坐标插值、Z-Buffer、Lambert 漫反射、PPM  
**项目仓库：** [soft_render3d](https://github.com/2027870784/soft_render3d)

本项目是一个从零实现的 C++ 软件 3D 渲染器。在不依赖 OpenGL、DirectX 等高级图形 API 的前提下，手写数学库、三角形网格、帧缓冲、光栅化器和基础渲染管线，完成从模型数据到像素图像输出的完整闭环。

## 项目背景

在学习计算机图形学过程中，很多概念如果只停留在公式和 API 调用层面，很难真正理解其工程含义。因此我希望通过软件渲染器把 MVP 变换、透视除法、重心坐标、深度测试、光照计算等核心流程完整实现一遍。

项目目标不是直接做一个高性能渲染器，而是通过尽可能透明的 C++ 实现，理解图形 API 背后隐藏的渲染管线细节。

## 核心功能

- **数学基础库**：实现 `Vector3`、`Vector4`、`Matrix4` 等基础类型，支持点乘、叉乘、归一化和矩阵运算。
- **几何数据建模**：使用 `Triangle` 和 `Mesh` 组织三角形、顶点颜色和法线信息。
- **MVP 渲染管线**：完成 Model、View、Projection 矩阵变换，以及透视除法和屏幕空间映射。
- **三角形光栅化**：基于屏幕空间包围盒、叉积判定和重心坐标插值完成三角形填充。
- **深度测试**：使用 Z-Buffer 处理像素遮挡关系，保证前后物体覆盖正确。
- **基础光照**：实现 Lambert 漫反射模型，并处理模型旋转后的法线变换问题。
- **图像输出**：将帧缓冲写出为 `.ppm` 图像，形成可观察的渲染结果。
- **工程组织**：使用 CMake 管理项目，将渲染核心与可执行入口拆分为更清晰的模块结构。

## 模块设计

项目采用较直接的模块划分，便于观察渲染管线中每个阶段的职责：

```text
soft_render3d/
├─ CMakeLists.txt
└─ renderer/
   ├─ math3d.hpp / math3d.cpp
   ├─ Triangle.hpp / Triangle.cpp
   ├─ mesh.hpp
   ├─ framebuffer.hpp / framebuffer.cpp
   ├─ rasterizer.hpp / rasterizer.cpp
   ├─ pipeline.hpp / pipeline.cpp
   └─ main.cpp
```

整体流程可以概括为：

```text
Mesh
-> Model / View / Projection
-> Clip Space
-> Perspective Divide
-> NDC
-> Screen Space
-> Rasterizer
-> Depth Test
-> Framebuffer
-> PPM
```

## 主要实现

### 1. 数学库与坐标变换

项目首先实现了向量和矩阵运算，为后续管线提供基础能力。`Vector4` 用于表达齐次坐标，其中 `w=1` 表示位置，`w=0` 表示方向。这个区别在法线变换中尤其重要，因为法线作为方向不应该受到平移影响。

在渲染管线中，顶点依次经过模型矩阵、视图矩阵和投影矩阵：

```text
MVP = Projection * View * Model
```

随后进行透视除法，将裁剪空间坐标转换到 NDC，再映射到屏幕像素坐标。

### 2. 三角形光栅化

光栅化阶段通过三角形屏幕空间包围盒减少遍历范围，再对包围盒内像素中心进行三角形内外判定。对位于三角形内部的像素，使用重心坐标插值深度和颜色，并交给帧缓冲进行深度测试。

这一阶段让我理解了三角形从几何数据变成像素的关键过程，也直观体会到 AABB 预筛对减少无效像素遍历的作用。

### 3. Z-Buffer 深度测试

帧缓冲中同时维护颜色缓冲和深度缓冲。每次写入像素前，先比较当前片元深度和已有深度，只有更靠近相机的片元才允许更新颜色。

项目中曾经出现过“先手动更新 z-buffer，再调用 setPixel 导致颜色无法写入”的问题。后续将深度测试和颜色写入统一收敛到 `setPixel()`，避免状态更新分散造成逻辑冲突。

### 4. Lambert 光照与法线变换

基础光照使用 Lambert 漫反射模型：

```text
intensity = max(0, dot(normal, light_dir))
```

当模型旋转后，如果只变换顶点而不变换法线，会出现几何已经旋转但光照仍按旧方向计算的问题。项目中通过将法线作为 `w=0` 的方向向量乘以模型矩阵，解决了旋转模型后光照不一致的问题。

### 5. CMake 工程化

项目使用 CMake 组织构建，并采用“核心库 + 可执行程序”的结构：

```cmake
add_library(renderer_core STATIC)
add_executable(soft_render3d main.cpp)
target_link_libraries(soft_render3d PRIVATE renderer_core)
```

这种组织方式让渲染核心逻辑和入口程序解耦，后续扩展测试、实时窗口或更多示例时更容易维护。

## 关键问题与解决方案

### 输出图像全黑

**问题：** 程序运行后生成图像全黑。  
**排查方向：** 相机位置、近裁剪面、透视除法、深度测试和颜色写入。  
**解决方案：** 调整相机位置，透视除法前检查 `clip.w`，并将深度测试和颜色写入统一放到 `setPixel()` 中处理。

### 输出速度过慢

**问题：** 一次输出图像耗时接近 20 秒。  
**原因：** 早期将 `saveAsPPM()` 放在三角形循环中，导致每处理一个三角形就完整写一次文本 PPM 文件。  
**解决方案：** 将图像输出移到整帧渲染结束后，只写出最终帧缓冲。

### 模型旋转后只剩单个亮面

**问题：** 模型顶点旋转后，其他面几乎全黑。  
**原因：** 顶点参与了模型变换，但法线仍停留在模型局部空间。  
**解决方案：** 在光照计算前将法线变换到世界空间，并保持方向向量 `w=0` 的语义。

### 屏幕空间插值不够准确

**问题：** 直接使用屏幕空间重心坐标插值颜色，在透视投影下会产生不严格的视觉结果。  
**解决方案：** 在实验分支中引入基于 `1/w` 的透视正确插值，使颜色过渡更符合透视投影关系。

## 项目亮点

- 从零实现最小可用软件渲染管线，覆盖数学库、几何处理、光栅化、深度测试、光照和图像输出。
- 不依赖高级图形 API，能够更直接地观察从模型到像素的每一步计算过程。
- 通过实际排查黑屏、深度写入、法线变换和输出性能问题，加深了对图形学工程细节的理解。
- 使用 CMake 进行工程组织，为后续测试、窗口渲染和多模型场景扩展提供基础。
- 在实验分支中探索实时窗口渲染、FPS 显示和多模型场景，验证渲染核心的扩展潜力。

## 项目成果

- 完成立方体网格的变换、投影、屏幕映射、三角形光栅化和深度测试。
- 实现基础 Lambert 漫反射光照，并处理模型旋转后的法线变换。
- 支持将渲染结果输出为 `.ppm` 图像。
- 初步建立 C++ 图形学项目的模块化组织方式。

## 个人收获

这个项目让我把图形学中抽象的概念放进了真实代码里。通过手写渲染管线，我更清楚地理解了坐标系约定、齐次坐标、透视除法、重心坐标、深度缓冲、法线变换和基础光照之间的关系。

相比直接调用成熟图形 API，从零实现软件渲染器更慢，也更容易出错；但正是这些错误暴露出了渲染管线中最值得理解的部分。
