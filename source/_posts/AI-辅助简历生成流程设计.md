个人简历最难维护的地方，不是第一次写出来，而是后续持续更新。每新增一个项目，如果要同时更新博客、项目页和多份简历，很容易出现信息重复和版本不一致。

因此，我在技术门户项目中设计了一套 AI 辅助简历生成流程。它的目标不是让 AI 完全代写简历，而是把已经写好的项目复盘转换成更结构化、更适合不同方向展示的简历内容。

# 基本思路

这个流程的输入不是零散的一句话，而是项目页和项目复盘文章。

项目页里已经包含了很多简历需要的信息：

- 项目名称
- 项目背景
- 技术栈
- 个人职责
- 核心功能
- 关键问题
- 项目成果

AI 更适合在这些已有材料上做提炼，而不是凭空生成经历。

整体流程可以设计为：

```text
项目 Markdown
-> 解析 front matter 和正文
-> 提取结构化项目数据
-> 按方向筛选和排序
-> 生成方向化简历页面
```

# front matter 的作用

为了让项目可以被程序识别，每个项目页都可以在 front matter 中加入结构化字段：

```yaml
resume: true
directions:
  - web
  - frontend
  - tooling
tech_stack:
  - Hexo
  - Markdown
  - GitHub Actions
role: 独立开发
weight: 90
```

这些字段解决了几个问题：

- `resume` 决定项目是否进入简历候选集
- `directions` 决定项目适合哪些方向
- `tech_stack` 让简历可以快速生成技能关键词
- `role` 保留个人职责
- `weight` 用于控制项目排序

这样一来，项目页既是展示页面，也是简历生成的数据源。

# AI 应该做什么

AI 在这个流程里最适合做内容提炼和表达转换。

例如项目页中可能有一段较长的描述：

```text
使用 Hexo 静态网站生成器搭建个人技术门户，使用 Markdown 管理技术文章、项目经历和在线简历，并通过 GitHub Actions 自动构建后部署到 GitHub Pages。
```

用于简历时，可以提炼成更短的成果表达：

```text
基于 Hexo 和 GitHub Actions 搭建个人技术门户，实现博客、项目经历和在线简历的 Markdown 化管理与自动部署。
```

AI 的价值在于把项目叙事转换成简历语言，同时尽量保留事实边界。

# 为什么需要本地降级

如果构建流程完全依赖 AI API，会有一个风险：API 不可用时，网站构建失败。

个人站点应该优先保证可构建。因此更稳妥的设计是：

```text
有 API Key -> 调用 AI 提炼
无 API Key 或调用失败 -> 使用本地规则提取
```

本地规则可以做得简单一些，例如：

- 读取项目标题
- 提取技术栈
- 截取项目概览中的前几句话
- 按 `directions` 和 `weight` 生成简历项目列表

这样即使 AI 不可用，简历页面仍然可以生成，只是表达质量可能没有 AI 提炼版本细致。

# 多方向简历

同一个项目可以在不同简历中突出不同侧重点。

例如“个人技术门户与自驱动简历系统”：

- 在 Web / 前端方向中，强调页面结构、响应式主题、搜索、评论和部署
- 在通用软件开发方向中，强调内容建模、CI/CD、配置管理和工程闭环
- 在 AI 工具方向中，强调项目数据结构化和自动简历生成

这种差异不适合手动复制多份内容长期维护。更合理的方式是把项目事实维护一次，再根据方向生成不同表述。

# 内容质量控制

AI 简历生成最需要控制的是真实性。

我给这个流程设定了几个约束：

- 不生成没有出现在项目材料中的技术栈
- 不虚构公司、实习、奖项和上线规模
- 不把设计方案描述成已经完整上线的功能
- 保留“探索、设计、原型、规划”等真实状态
- 对关键项目仍然人工复核

这些约束可以避免简历内容变得夸张。对个人项目来说，真实、具体、可追溯比堆砌关键词更重要。

# 后续扩展

这套流程还可以继续完善：

- 为项目页设计统一模板，提升提取稳定性
- 生成 `projects.generated.json` 作为中间数据
- 为不同方向配置关键词权重
- 增加简历打印样式，支持 PDF 导出
- 在构建日志中标注 AI 生成或本地降级来源

# 总结

AI 辅助简历生成的重点不是省掉写作，而是降低重复维护成本。

当博客、项目页和简历都在同一个内容系统里，项目经历就可以被多次复用：文章记录过程，项目页展示结果，简历提炼能力。AI 参与其中，最合适的位置是帮助提炼和重写，而不是替代真实项目本身。
