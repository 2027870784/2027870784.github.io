个人技术门户如果只是能访问，距离“可以用于项目验收展示”还有一段距离。验收时老师或同学通常不会逐行阅读代码，而是会先看站点是否有完整入口、内容是否成体系、项目成果是否能被快速理解。因此，这次我把站点内容从“有几篇文章”整理成了更清晰的展示结构。

这篇文章记录这次内容结构优化的思路。

# 展示目标

这次优化的目标很明确：让访问者进入网站后，能够快速看到这个项目做了什么、为什么做、用了哪些技术、有哪些可展示成果。

围绕这个目标，站点需要具备几个核心内容块：

- 博客文章：展示学习过程、工程复盘和技术思考
- 项目经历：用结构化方式说明每个项目的背景、职责、功能和成果
- 在线简历：把项目经历转换成更适合求职或验收展示的能力说明
- 关于页面：解释站点定位，避免只是一个零散文章集合

这些内容不是彼此独立的。博客负责讲过程，项目页负责讲结果，简历负责讲能力，关于页负责讲整体定位。

# 为什么不只做首页

很多展示型网站会把全部信息塞到首页，但个人技术门户更适合内容分层。

如果所有内容都堆在首页，短期看起来信息很多，长期维护会变得困难。新增一个项目时，需要同时改首页、简历、项目介绍和文章摘要，容易出现内容不一致。

现在的结构更接近内容资产库：

```text
source/
  _posts/      # 博客文章
  projects/    # 项目详情
  resume/      # 多版本简历
  about/       # 站点说明
```

每一类内容有自己的职责。后续新增项目时，可以先写项目详情页，再补一篇复盘文章，最后把它整理进对应方向的简历。

# 博客内容怎么服务验收

博客文章不只是“凑数量”。更适合验收展示的博客应该能回答几个问题：

- 这个项目为什么要做
- 技术选型是怎么来的
- 中间遇到了哪些问题
- 最终做到了什么程度
- 后续还能怎么扩展

例如 Hexo 站点搭建文章可以说明部署链路，C++ 软件渲染器文章可以说明技术深度，AI 简历生成文章可以说明扩展规划。这样博客区就不只是文章列表，而是项目成长过程的记录。

# 项目页的组织方式

项目经历页需要比博客更结构化。博客可以有叙事节奏，项目页则应该方便快速扫描。

我采用了比较稳定的模板：

```text
项目概览
项目背景
核心功能
主要工作
技术实现
问题与解决方案
项目亮点
项目成果
个人收获
```

这个模板适合个人项目，也适合课程验收。访问者不需要读完整篇文章，也能通过标题快速定位关键信息。

# 简历页的角色

简历页不是把项目页复制一遍，而是把项目经历重新组织成“能力证明”。

同一个项目可以服务不同方向：

- C++ / 图形学方向强调渲染管线、数学库、光栅化和调试能力
- Web / 前端方向强调静态站点、页面结构、部署和内容组织
- 通用软件开发方向强调工程拆分、版本管理、自动化和问题排查

因此，简历入口页采用“多个版本”的形式，而不是只保留一份通用简历。这样后续扩展也更自然。

# 展示动线

用于验收时，我推荐按下面的顺序展示：

1. 先打开首页，说明这是一个技术门户，不只是普通博客。
2. 进入项目页，展示已有项目列表和每个项目的详情结构。
3. 打开 C++ 渲染器项目，说明技术深度和完整实现闭环。
4. 打开技术门户项目，说明这个网站本身也是项目成果。
5. 进入简历页，展示同一批项目如何沉淀为多方向简历。
6. 最后打开博客归档，说明项目过程有持续记录。

这条动线可以在较短时间内覆盖“功能、内容、技术、成果、可维护性”几个验收重点。

# 这次优化的收获

这次内容整理让我意识到，个人网站真正重要的不是页面数量，而是内容之间是否形成关系。

博客、项目、简历和关于页面如果彼此割裂，访问者需要自己拼出全貌；如果结构清晰，它们就能共同说明一个完整项目：我做了什么，怎么做的，做完后沉淀出了什么。

对于课程项目或个人作品集来说，这种结构比单纯追求视觉效果更重要。它能让项目展示更稳定，也让后续维护更轻松。
