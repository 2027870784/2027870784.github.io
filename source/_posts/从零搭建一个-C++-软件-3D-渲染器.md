这个项目的目标很朴素：不用 OpenGL、DirectX 或现成渲染 API，自己用 C++ 写一条最小可用的软件 3D 渲染管线。换句话说，我想把课堂或书本里的那些概念，比如齐次坐标、MVP 变换、透视除法、重心坐标、深度缓冲和 Lambert 光照，真正落到代码里，再让它们生成一张可见的图像

项目最后形成了一个名为 `soft_render3d` 的小型软件渲染器。当前主线可以完成立方体网格的变换、投影、屏幕映射、三角形光栅化、深度测试、简单漫反射光照，并导出 PPM 图像；在 `temp_advance` 分支中，还进一步探索了实时窗口渲染、FPS 显示、金字塔和八面体等多模型场景

# 项目结构

项目采用了比较直接的模块划分：

```text
soft_render3d/
├─ CMakeLists.txt
└─ renderer/
   ├─ CMakeLists.txt
   ├─ math3d.hpp / math3d.cpp
   ├─ Triangle.hpp / Triangle.cpp
   ├─ mesh.hpp
   ├─ framebuffer.hpp / framebuffer.cpp
   ├─ rasterizer.hpp / rasterizer.cpp
   ├─ pipeline.hpp / pipeline.cpp
   └─ main.cpp
```

其中 `math3d` 是整个项目的基础，提供 `Vector3`、`Vector4` 和 `Matrix4`；`Triangle` 和 `Mesh` 描述几何数据；`Framebuffer` 保存颜色缓冲和深度缓冲；`Rasterizer` 负责把屏幕空间三角形填成像素；`Pipeline` 则把模型数据串成完整的渲染流程

整体管线可以概括为：

```text
Mesh
 -> Model / View / Projection
 -> Clip Space
 -> Perspective Divide
 -> NDC
 -> Screen Space
 -> Rasterizer
 -> Depth Test
 -> Framebuffer
 -> PPM / Window
```

这条流程看起来不长，但每一步都藏着图形学里很容易踩坑的细节

# 第一阶段：先把数学地基打稳

项目最早从 `math3d` 开始。最初只是实现三维向量、四维向量和 4x4 矩阵，后来逐步补上点乘、叉乘、归一化、齐次坐标转换、矩阵与向量乘法、矩阵与矩阵乘法等能力

这个阶段最大的收获是：图形学里的向量类型不能只当作普通数组来看。`Vector3` 通常用于方向、法线、颜色等三分量数据，而 `Vector4` 承担齐次坐标语义，`w=1` 表示位置，`w=0` 表示方向。这个区别后来在法线变换时非常关键

我也在这个阶段把头文件和实现文件拆开，让 `math3d.hpp` 尽量只保留声明，把函数定义放到 `math3d.cpp` 中。虽然项目规模还不大，但这一步让接口和实现的边界更清晰，也让后续 CMake 组织更自然

# 第二阶段：从三角形到立方体

有了基础数学库之后，下一步是定义可以被渲染的几何对象。项目先加入了 `Triangle` 类，用三个 `Vector4` 顶点、三个顶点颜色和一个面法线来描述最小绘制单元。随后又加入 `Mesh`，用一组顶点、三角形、法线和颜色组织模型

立方体的实现看似简单，但这里其实第一次遇到了“图形学不是只要连线就行”的问题。一个标准立方体需要 6 个面，每个面 2 个三角形，总共 12 个三角形；每个面的三角形顶点绕序要一致，法线也要朝外

早期版本中，部分三角形索引并不在同一个立方体面上，有些三角形会斜穿过模型内部，法线也没有按面对应。修正之后，`createCube()` 明确约定：

- `+X` 是模型右侧
- `+Y` 是模型上方
- `+Z` 是模型正面
- 项目整体使用右手系
- 当前相机放在 `+Z` 方向，看向原点，即相机朝 `-Z` 观察

这个约定后来帮助我解释了很多现象，比如为什么模型正面法线是 `(0, 0, 1)`，以及为什么相机在 `+Z` 上却是朝 `-Z` 看

# 第三阶段：帧缓冲、深度缓冲与第一张图

为了真的输出图像，项目加入了 `Framebuffer`。它内部维护两块缓冲：

- `colors`：每个像素的颜色
- `z_buffer`：每个像素当前记录的最近深度

渲染前先用背景色清空颜色缓冲，并把深度缓冲初始化为无穷大。写像素时只有当新片元的深度更近，才更新颜色和深度。这个小小的判断就是最基础的遮挡关系

输出格式一开始选择了 PPM，主要是因为它极其简单，不需要额外图片库。当前实现使用的是 `P3` 文本格式，优点是好调试，缺点也很明显：每个像素都以 ASCII 文本写盘，分辨率稍高时 I/O 成本会很大

这也引出了后面一次很典型的性能问题

# 第四阶段：光栅化和渲染管线

真正让三角形变成像素的是 `Rasterizer`。它做了几件事：

1. 计算三角形在屏幕空间的包围盒
2. 遍历包围盒内的像素中心点
3. 用叉积判断点是否在三角形内部
4. 计算重心坐标
5. 插值深度和颜色
6. 交给帧缓冲做深度测试和像素写入

`Pipeline` 则负责更上游的几何变换。它先构造视图矩阵和投影矩阵，再计算：

```cpp
MVP = projection * view * model;
```

顶点经过 MVP 后进入裁剪空间，再进行透视除法得到 NDC，最后映射到屏幕坐标。屏幕映射中特别要注意 `y` 方向：NDC 里 `+Y` 向上，而屏幕像素坐标通常 `y` 向下，所以需要做一次翻转

这个阶段还加入了最简单的 Lambert 漫反射：

```cpp
intensity = max(0, normal dot light_dir);
```

它足够简单，也足够暴露问题

# 第五阶段：一次黑图和 20 秒输出的排查

项目中最有价值的一次调试，是“程序运行后输出一张图大约要 20 秒，而且结果全黑”

慢的原因很直接：早期 `saveAsPPM()` 被放进了三角形循环里。立方体有 12 个三角形，这意味着每处理一个三角形就完整写一次 `800x600` 的 PPM 文本图像。一次渲染不是写一帧，而是反复写十几帧，而且还是文本格式，耗时自然被 I/O 放大

全黑的问题更有层次。排查时先怀疑光照，但很快发现主因不止一个：

- 相机一度和立方体都在原点附近，几何穿过相机或近裁剪面
- 项目还没有真正的视锥体裁剪，跨平面的三角形不会被正确切分
- 透视除法前的合法性应该检查 `clip.w()`，而不是检查 `clip.z()`
- `w_division()` 在 `w` 接近 0 时直接伪造默认点，会掩盖上游错误
- 光栅化阶段曾经先手动更新 `z_buffer`，再调用 `setPixel()`，而 `setPixel()` 内部又做了一次深度判断，导致颜色永远无法写入

最后修正的核心思路是：让 `setPixel()` 统一负责深度测试和颜色写入；透视除法前检查 `w`；把相机放到物体前方，例如 `(0, 0, 3)`；并把 PPM 输出移出三角形循环

这次调试让我意识到，黑屏不一定是“没有画”，也可能是几何空间、深度状态、颜色范围和输出时机共同叠加出的结果

# 第六阶段：旋转模型后，为什么只剩一个红面

为了让静态立方体更有 3D 体积感，我在 `main.cpp` 中把 `model_matrix` 从单位矩阵改成绕 `Y` 轴和 `X` 轴的组合旋转。这样输出图中应该能看到多个面

但实际结果一开始只看到一个红色面。这个现象很有意思：几何确实旋转了，因为红色面的形状变了；但其它面几乎都黑了

原因是顶点参与了模型矩阵变换，法线却仍然停留在模型局部空间。也就是说：

```text
顶点转了，光照判断没跟着转。
```

正确做法是在光照前把法线也变到世界空间。由于当时 `model_matrix` 只有旋转，没有非等比缩放，所以可以把法线当作 `w=0` 的方向向量乘以模型矩阵：

```cpp
Vector4 normal4 = triangle.getNormal().toVector4_w0();
Vector3 normal_world = (model_matrix * normal4).toVector3().normalize();
```

这里 `w=0` 很关键，因为法线是方向，不应该受平移影响。如果未来支持非等比缩放，则需要引入更标准的法线矩阵，也就是模型矩阵左上 3x3 的逆转置

# 第七阶段：从屏幕线性插值到透视正确插值

在实现顶点颜色渐变后，又遇到了一个典型问题：直接用屏幕空间重心坐标插值颜色，在透视投影下并不严格正确。因为重心坐标是在屏幕空间算出来的，三角形经过投影后已经发生了非线性变形

项目在实验分支中把颜色插值改成了透视正确插值。核心做法是保留每个屏幕顶点的原始 `clip.w`，插值属性时按 `1/w` 修正：

```cpp
Vector3 final_color =
    (colors[0] * (alpha / a.w()) +
     colors[1] * (beta / b.w()) +
     colors[2] * (gamma / c.w())) * w_reciprocal;
```

深度插值本来就已经朝这个方向靠拢，颜色也统一到同一套公式后，倾斜面上的过渡会更符合透视投影下的视觉效果

# 第八阶段：CMake 工程化

项目一开始只是源码文件的集合，后来加入了两层 CMake：

- 根目录 `CMakeLists.txt` 负责工程名、版本、语言和全局选项
- `renderer/CMakeLists.txt` 负责具体 target

当前采用的是“核心库 + 可执行程序”的组织方式：

```cmake
add_library(renderer_core STATIC)
add_executable(soft_render3d main.cpp)
target_link_libraries(soft_render3d PRIVATE renderer_core)
```

这样做的好处是，渲染核心和入口程序解耦。后续要加测试、示例、实时窗口、不同前端或命令行参数时，不需要把所有逻辑都塞进一个 `main.cpp`

这个过程中还修正了 `pipline` 到 `pipeline` 的拼写问题，并补上了一些更符合跨编译器要求的头文件依赖。用 GNU `g++` 编译时，编译器对缺失标准头文件更严格，这反过来也帮助项目暴露了依赖不完整的问题

# 实验分支：实时渲染和多模型场景

主线当前仍以单帧 PPM 输出为主，但 `temp_advance` 分支做了更激进的探索：

- 使用 Win32 窗口和 GDI，把 `Framebuffer` 直接拷贝到窗口
- 通过主循环按时间更新模型旋转
- 在左上角绘制 `FPS`
- 新增金字塔和八面体网格
- 把多个模型先变换并合并成一个场景网格，再一次性提交给管线渲染

这条分支的意义不是“功能已经完全成熟”，而是验证了一件事：前面搭出来的渲染核心确实可以从离线输出图片，继续演进到实时显示

# AI 协作给我的帮助

这个项目中我大量使用 AI 作为代码审查和图形学概念校对工具。它最有帮助的地方不是直接生成代码，而是在我遇到现象时帮我拆问题：

- 只看到一个红面时，反推到“顶点转了但法线没转”
- 黑图时，按相机、裁剪、透视除法、深度写入、颜色范围逐层排查
- 配置 CMake 时，帮助选择“核心库 + 可执行程序”的结构
- 检查立方体三角形索引时，把“是否在同一平面”和“法线是否朝外”作为验证标准

这种协作方式很适合图形学项目。因为渲染问题往往不是单点错误，而是多个数学约定一起作用。AI 可以帮我把现象拆成可验证的假设，而真正的判断仍然要回到代码、构建和输出结果

# 当前不足

这个项目还只是一个学习型软件渲染器，距离完整渲染器还有不少路：

- 还没有真正的视锥体裁剪。当前只是在透视除法前检查 `w`，并在屏幕空间限制光栅化包围盒
- 没有背面剔除
- 光照仍是非常简单的按面 Lambert 漫反射
- 颜色范围约定还可以继续统一，最好内部全部使用 `[0,1]`
- PPM `P3` 文本输出适合调试，但不适合性能要求更高的场景
- 缺少自动化测试，尤其是数学库、矩阵乘法、透视除法和光栅化边界条件
- 法线变换还没有支持非等比缩放下的逆转置矩阵

如果继续推进，我会优先补 `near plane clipping`，因为它对稳定性影响最大；其次是背面剔除、二进制 PPM 或实时窗口路径，以及给数学库补测试

# 总结

这次项目最有价值的地方，不是最终渲染出了一个立方体，而是把“从模型到像素”的每个环节都踩了一遍

写一个软件渲染器会迫使我正视很多平时被图形 API 隐藏起来的细节：坐标系怎么约定，`w` 到底代表什么，为什么法线不能和顶点一样随便处理，为什么屏幕空间插值会出错，为什么一张黑图背后可能同时有相机、裁剪和深度缓冲的问题

项目目前还很小，但它已经完成了一个重要的闭环：从数学库出发，到几何建模、MVP 变换、光栅化、深度测试、光照和图像输出。对我来说，这比直接调用现成 API 更慢，但也更接近图形学真正工作的地方
